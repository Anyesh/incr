How incr works

incr is an incremental computation library. Instead of recomputing everything when one thing changes, it tracks which computations depend on which inputs and only reruns what's affected.

The computation graph

Every value in incr lives in a node. Inputs hold raw values that you set directly; queries compute their value from other nodes. incr tracks the edges between them automatically, forming a directed acyclic graph.

Inputs hold values you set directly. Queries compute from their dependencies. incr tracks the edges automatically, so you never have to declare "area depends on width" by hand.

Change propagation

When you call rt.set(width, 12), incr doesn't immediately recompute anything. It marks the input as dirty and follows the edges forward, marking every dependent node dirty too. The actual recomputation only happens when you read a result with rt.get(area).

done

Dirty marks flow forward along edges. Recomputation flows backward from whatever you read. This lazy evaluation means nodes that nobody reads are never recomputed at all.

Early cutoff

If a node recomputes and produces the same value it already had, propagation stops right there. Downstream nodes never see the change, so they don't recompute. This lets incr skip entire subtrees of the graph when an upstream change turns out to be a no-op.

Incremental collections

A collection in incr is a set of elements piped through operators like filter, map, and reduce. When you insert or delete a single element, only that element flows through the pipeline. The engine doesn't re-examine anything that hasn't changed.

Each pipeline stage does O(1) work per changed element. Insert one item into a 100K-element collection and every stage touches exactly one element, not 100,000.
